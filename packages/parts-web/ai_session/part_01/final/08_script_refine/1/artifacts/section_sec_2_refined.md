# 传统临床决策支持系统（CDSS）的核心困境

在大型语言模型出现之前，传统的临床决策支持系统（CDSS）在实际应用中普遍面临失灵的困境，其根源可归结为以下几大核心挑战。

## 1. 数据之困：异构、低质与提取难题

CDSS 需要对接医院内部多样化的业务系统，导致数据层面存在严峻挑战。

- **数据来源异构**：数据分散于多个独立的业务系统中，例如：
  - 检验信息系统（LIS）
  - 放射学信息系统（RIS）
  - 医学影像存档与通信系统（PACS）
  - 护理系统
  - 医嘱系统
- **标准与协同性缺失**：不同厂商开发的系统遵循不同标准，数据之间缺乏有效的协同与整合机制。
- **质量参差不齐**：系统中的数据混合了结构化与非结构化信息，整体质量难以保证。

这些问题直接导致了数据特征提取的巨大困难，尤其是在自然语言处理（NLP）、光学字符识别（OCR）以及数据级提取等关键环节。

## 2. 算法与协同之困：静态模型与高耦合

多变的临床场景与传统算法架构之间的矛盾，催生了算法和系统协同的困境。

- **场景动态多变**：临床决策场景会因国家政策更新、最新科研文献发布的诊疗路径等因素而持续动态变化。
- **静态模型应对**：传统的做法是为每一个特定场景开发一个独立的、静态的规则引擎。这种“一个场景，一个引擎”的模式导致模型与业务逻辑紧密绑定。
- **协同成本剧增**：随着场景增多，系统内部会衍生出大量的“小模型”和独立的后端服务。这不仅导致维护成本线性增长，还使得服务间的耦合度越来越高，系统架构如同“堆烟囱”般日益臃肿复杂。

## 3. 环境之困：安全要求与算力限制

医院的运行环境为 CDSS 的部署和运行带来了独特的限制。

- **高安全与合规要求**：患者数据是核心敏感信息，必须在严格的内网环境下运行，以满足数据安全与合规要求。
- **算力资源有限**：与高安全要求相对的是，医院内部的服务器硬件配置和算力资源通常有限。

这种“高要求”与“低资源”的矛盾，使得在算法模型推理、业务系统运行之间进行有效的资源协调变得极为困难，从而引发一系列性能和稳定性问题。
